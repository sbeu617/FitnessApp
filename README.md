# Fitness App — тестовое задание

Android-приложение на Kotlin с двумя экранами: список тренировок с поиском/фильтрацией и экран видеоплеера с деталями тренировки.

## Стек

- **Язык:** Kotlin
- **UI:** View-based (XML), Material 3, без Jetpack Compose
- **Архитектура:** MVVM, навигация — Navigation Component + Safe Args (single-Activity)
- **Сеть:** Retrofit + Gson
- **Видео:** ExoPlayer / Media3
- **Асинхронность:** Kotlin Coroutines + StateFlow

## Архитектура

Проект разделён на три слоя без отдельного domain/UseCase слоя — он не добавляет ценности при таком объёме бизнес-логики (два эндпоинта, без сложных сценариев между несколькими источниками данных):

```
data/           — DTO, Retrofit-интерфейс, реализация репозитория, мапперы DTO → domain
domain/         — чистые модели (Workout, WorkoutType, VideoInfo) и интерфейс репозитория
presentation/   — Fragment + ViewModel + sealed UiState на каждый экран
```

ViewModel зависит от интерфейса `WorkoutRepository`, а не от конкретной реализации — это даёт тестируемость без необходимости заводить полноценный DI-фреймворк (Hilt/Koin не подключены, фабрики ViewModel собираются вручную через `ViewModelProvider.Factory`).

Состояния экранов выражены через `sealed class UiState` (`Loading` / `Success` / `Empty` / `Error`), собираются в один `StateFlow` через `combine` (на экране списка — данные + поисковый запрос + выбранный фильтр объединяются реактивно).

Навигация — Fragment + единственная Activity-хост, вместо изначально рассматривавшегся варианта с двумя Activity. Данные о тренировке передаются между экранами через Safe Args (`Workout` сделан `Parcelable`), чтобы не делать повторный запрос к `/get_workouts` на экране плеера.

## Решения по несоответствиям API и ТЗ

В процессе работы со Swagger-схемой и реальным тестовым сервером обнаружился ряд расхождений. Ниже — что было найдено и как обработано.

### Типы тренировок

В ТЗ указаны текстовые категории (Кардио, Силовые, Растяжка), в Swagger-схеме поле `type` — `integer` с описанием `1 - тренировка, 2 - эфир, 3 - комплекс`. Эти два описания друг другу не соответствуют по смыслу. Решено ориентироваться на формулировки из текста ТЗ и явно замапить `1 → Кардио, 2 → Силовые, 3 → Растяжка` (`WorkoutType.fromApiValue`) — поскольку ТЗ описывает ожидаемый пользовательский интерфейс, а числовые значения в схеме выглядят как условные идентификаторы категорий, а не как точное описание их смысла.

### `description: null`

У части тренировок поле `description` приходит как `null` (это есть и в самом примере из схемы). Обрабатывается при маппинге DTO → domain: при `null` подставляется текст "Описание отсутствует".

### `duration` — непредсказуемый тип

В схеме `duration` — строка вида `"30 минут"`. На реальном сервере это поле в большинстве случаев приходит как число без единиц измерения (`14`, `45`, `57`...), а в нескольких записях — строка-заглушка `"workout"`, не являющаяся числом. Поле в DTO оставлено как `String` (любое значение безопасно сериализуется в строку без падения парсинга), а при маппинге в domain-модель применяется безопасный парсинг: если строку удаётся привести к числу — формируется фраза с корректным русским склонением ("14 минут", "2 минуты", "1 минута"), иначе — выводится "Длительность не указана".

### `link` — абсолютный или относительный путь

Пример в схеме показывает абсолютный URL (`https://archive.org/...`), но на практике сервер также отдаёт **относительные** пути (`/example-05.mp4`). При маппинге `VideoDto → VideoInfo` относительный путь резолвится в полный URL через базовый адрес сервера; абсолютные ссылки используются как есть.

## Качество видео

API отдаёт единственный mp4-файл без alternative renditions (HLS/DASH-манифеста с несколькими битрейтами/разрешениями) — то есть данных для переключения качества на реальном источнике физически не существует.

Реализован честный механизм без фейковой имитации:

- Кнопка качества встроена в контролы плеера (появляется и скрывается синхронно с остальными элементами управления `PlayerView`).
- При получении треков (`Player.Listener.onTracksChanged`) определяется, содержит ли источник несколько вариантов видеодорожки.
- Если вариантов несколько (adaptive-поток) — кнопка реально переключает `TrackSelectionParameters.setMaxVideoSize` между 240p/480p/720p/Авто.
- Если вариант один (обычный mp4, как у всех `/get_video` ссылок в этом API) — нажатие показывает пользователю понятное сообщение о том, что для этого видео доступно только одно качество, вместо притворного переключения.

Механизм протестирован на adaptive-источнике (публичный HLS-поток) и подтверждённо переключает качество там, где это технически возможно.

## Перемотка и скорость воспроизведения

Обе функции работают через встроенный интерфейс Media3 `PlayerView`: перемотка — через стандартный `DefaultTimeBar`, скорость воспроизведения — через встроенное меню настроек плеера (иконка шестерёнки). Дополнительный кастомный UI не потребовался — Media3 покрывает эти требования из коробки.

## Состояния экранов

Оба экрана (список и плеер) реализуют единый паттерн `Loading / Success / Empty (только список) / Error`, управляемый через `StateFlow<UiState>` во ViewModel и `repeatOnLifecycle` во Fragment. Ошибки сети и парсинга оборачиваются в `Result`, не приводя к падению приложения.
